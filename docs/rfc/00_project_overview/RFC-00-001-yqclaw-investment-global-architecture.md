## 1. 执行摘要
本RFC为 YQClaw Investment 项目（AI智能投资交易系统）唯一总架构规范，定义系统定位、整体架构、完整模块划分、依赖关系、数据流转、AI实装规则、开发标准与**全模块八大标准化体系**。项目业务功能涵盖了技术基础设施、数据底座、金融投资交易领域知识、组合管理及分析、策略研发及实盘、交易支持、投研分析、丰富的报告等，并对目录、代码、数据、接口、AI开发、异常容错、版本迭代、环境配置全维度规范。

本文档是所有子模块RFC编写、AI工具自动化开发实装的唯一依据，所有子模块RFC及代码实装必须严格遵循本总纲规范，保障系统全局统一、架构一致、可运维、可复盘、可自动化落地、长期可迭代。

## 2. 背景与动机
### 2.1 业务痛点
当下个人投资存在信息碎片化、投研效率低下、交易决策情绪化、组合风控滞后、策略无法系统化研发回测、缺少标准化监测与定制报告体系，数据不通畅、流程无法闭环、无AI智能辅助决策等核心问题。

同时，传统开发模式无统一工程规范，存在目录杂乱、代码风格不统一、数据口径混乱、接口不标准、AI生成代码差异化严重、无统一容错降级机制、迭代无追溯、配置随意硬编码等问题，导致AI编码工具无法精准落地、模块对接冲突、后期维护迭代成本极高，且量化系统对**数据一致性、流程可追溯、运行可复盘、风险可控**的核心诉求无法满足。

### 2.2 项目定位
打造自主可控，AI Agent全自动驱动的闭环智能投资交易系统，覆盖多品类资产投研、组合管理、策略研发回测、自动化交易、定制化报告、风险控制、投资知识沉淀、数据一体化处理全流程。

完全遵循**RFC先行、规范统一、AI实装跟进**的顶级AI工程化开发模式，配套全维度标准化工程体系，打造个人专属**标准化、自动化、可复盘、高稳健**的智能投资体系。

### 2.3 规范建设价值
通过建立全模块统一标准化体系，彻底解决项目开发杂乱、AI落地偏差、模块耦合、数据不一致、运维无依据、迭代无追溯的核心问题，实现**架构统一、代码统一、数据统一、行为统一、运维统一**，适配多AI工具协同开发，保障系统长期稳定迭代。

## 3. 目标与非目标
### 3.1 核心目标
- [ ] 定义YQClaw Investment系统完整模块化架构，新增策略模块、报告模块，划分清晰模块边界
- [ ] 制定全系统统一RFC编写规范、模块命名、数据、接口、AI实装标准
- [ ] 建立**八大全局标准化体系**，实现全模块工程规范统一
- [ ] 明确模块依赖、数据流转，AI Agent全局调度逻辑
- [ ] 所有子模块RFC必须继承本总纲规范，一一对应、严格对齐
- [ ] 完全适配AI工具全自动实装，零架构偏差、零代码混乱、零规范冲突
- [ ] 实现系统可追溯、可复盘、可容错、可扩展、可运维

### 3.2 非目标
- [ ] 不涉及对外经营性资管、理财服务，仅限个人专业投资使用
- [ ] 不设计冗余架构、不过度技术选型，聚焦投资业务核心，不做无效标准化堆砌
- [ ] 不接入第三方商用运维、日志、监控平台，保持项目轻量化、自主可控

## 4. 全局模块完整架构与职责
### 4.1 模块编号&amp;名称&amp;核心职责
00_project_overview：项目全局总架构，统一规范，全局约束、标准化体系定义
01_app：PC/移动端/H5 全端前后端应用与用户交互
02_common：全项目通用金融工具，公共计算、基础能力组件、全局工具封装、统一规范落地载体
03_data：市场数据采集、清洗、Pipeline，标准数据接口、全链路数据标准化落地
04_knowledge：投资理念、投研逻辑、策略知识库、领域知识图谱
05_portfolio：可投池管理、组合构建、持仓分析，调仓、业绩归因
06_strategy：各类量化/主观策略研发，历史回测、参数优化，对接模拟盘与实盘交易
07_trading：交易通道对接、订单管理、实盘/模拟盘执行、交易风控清算
08_research：股票/基金/ETF/Crypto资产研究，多因子评分、机构/卖方/牛散/社区投研、金融舆情
09_reports：资产监测报告、组合风控报告、每日/每周常规监测报表、个性化定制投研报告，自动导出归档
10_infra：系统底座，AI Agent调度、权限，日志、监控、部署运维、底层架构、全局规范落地

### 4.2 全局模块依赖规则
1. 上层模块可调用下层模块能力，下层不可反向调用上层，严格单向依赖
2. 同级模块相互独立，通过标准接口通信，禁止循环依赖、跨模块硬耦合
3. Strategy策略模块依赖数据、知识、组合模块，向下对接Trading交易模块
4. Reports报告模块依赖投研、组合、策略、数据模块，统一生成各类报表
5. 所有模块统一依赖common全局基础能力，统一遵循全局标准化规范

## 5. 全局八大标准化体系（全模块强制遵守）
本章节为**项目最高强制规范**，所有子模块设计、RFC编写、AI代码实装、迭代优化必须100%遵循，禁止私自变更、非标实现。

### 5.1 目录与文件命名标准化
#### 5.1.1 核心规则
- 全局目录、文件夹：纯小写+下划线命名，无驼峰、无中文、无特殊字符、无空格
- 业务代码文件：统一遵循 `{模块名}_{功能名}.py` 语义化命名
- RFC文档文件：统一遵循 `RFC-{YY}-{NNN}-{模块名}-{简述}.md` 标准编号命名
- 配置文件、日志文件、静态资源、脚本文件统一范式，禁止随机命名

#### 5.1.2 落地约束
- AI生成代码、新增目录必须严格对齐全局结构，禁止私自新建层级、乱放文件
- 所有文件命名语义唯一，见名知意，杜绝模糊命名（test、temp、new等非标命名）
- 项目全生命周期目录结构固定，新增功能必须在对应模块内扩展，不破坏全局结构

### 5.2 代码工程结构与风格标准化
#### 5.2.1 核心规则
- 全局严格遵循 Python PEP8 工业级编码规范
- 所有业务模块统一内部结构：`config / core / utils / model / service`，层级统一
- 类、函数职责单一，禁止超大函数、超大类、冗余代码
- 统一注释规范：核心逻辑注释**设计原因+实现逻辑**，而非仅注释代码行为

#### 5.2.2 落地约束
- 所有AI工具生成代码必须统一风格，多AI输出观感、结构、范式完全一致
- 禁止模块自定义编码风格、私自新增非标目录结构
- 核心功能必须补充简洁单元测试，代码可复用、可维护、可迭代

### 5.3 全链路数据标准化（量化核心强制规范）
#### 5.3.1 核心规则
- 金融字段统一：时间、价格、收益率、仓位、市值、波动率、回撤等核心字段**全局唯一命名**
- 数据类型统一：浮点精度、时间格式、空值定义、数值单位全局统一
- 数据流转统一：严格遵循「数据层→知识层→投研层→策略层→交易层→应用层」单向流转
- 数据运维统一：缓存规则、清洗规则、校验规则、降级兜底规则全模块一致

#### 5.3.2 落地约束
- 禁止各模块自定义时间格式、数据精度、统计口径，杜绝数据对账偏差、回测失真
- 所有数据结构新增、修改、废弃必须更新对应RFC，禁止无文档私自改结构
- 量化核心数据强制脱敏、校验，保证数据一致性、准确性、可复盘性

### 5.4 模块接口与调用标准化
#### 5.4.1 核心规则
- 全局统一接口请求、响应、异常返回格式
- 全局统一错误码、状态码、提示文案规范
- 模块间严格分层调用，只允许上层调用下层、同级标准化互通，禁止跨层乱调用、循环依赖

#### 5.4.2 落地约束
- 所有接口必须包含参数校验、异常捕获、日志埋点
- AI生成接口逻辑必须对齐全局契约，禁止自定义返回结构、错误规则
- 所有接口变更必须走RFC迭代，保证全模块对接兼容

### 5.5 多平台AI实装开发标准化
#### 5.5.1 核心规则
适配 OpenClaw、Claude Code、Gemini 全AI平台，统一开发规则：
- AI权限规则统一：明确可直接执行、需先询问、绝对禁止的操作
- AI代码生成范式统一：日志埋点、异常捕获、注释、单元测试、代码简洁度统一
- AI任务拆分统一：单指令单模块、单功能，禁止跨模块大范围修改
- AI迭代逻辑统一：**先更新RFC规范，再落地代码实现**

#### 5.5.2 落地约束
- AI禁止自由发挥、过度设计、冗余编码、偏离RFC设计
- AI禁止硬编码参数、密钥、路径等可变配置
- 所有AI开发成果必须与RFC文档1:1映射，无偏差落地

### 5.6 异常、容错与降级体系标准化
#### 5.6.1 核心规则
- 异常分类标准化：统一分为数据异常、策略异常、交易异常、业务异常、系统异常、AI执行异常
- 异常处理标准化：统一捕获范式、报错输出、堆栈打印、日志记录规则
- 降级策略标准化：全局统一兜底逻辑，数据兜底、策略暂停、交易阻断、报告降级、服务容错
- 日志规范标准化：全项目禁止使用print，统一使用 common 全局日志工具，详见5.6.3

#### 5.6.2 落地约束
- 所有核心业务必须具备降级方案，异常不阻断核心流程或可控阻断
- 交易、策略等高风险场景，异常优先风控、优先保资产安全
- 所有异常必须可追溯、可复盘、可定位根因

#### 5.6.3 全局日志统一管理规范
- **目录规范**：所有模块日志统一存放于 `workspace-yquant/logs/` 目录，按模块分文件夹隔离：`logs/system/`、`logs/app/`、`logs/common/`、`logs/data/`、`logs/knowledge/`、`logs/portfolio/`、`logs/strategy/`、`logs/trading/`、`logs/research/`、`logs/reports/`
- **命名规范**：`{模块名}_{YYYYMMDD}.log`，按天自动切割
- **格式规范**：`[年-月-日 时:分:秒] [模块名] [日志级别] [日志内容]`
- **级别规范**：DEBUG、INFO、WARN、ERROR、ORDER（交易专属）
- **强制约束**：全项目禁止使用print；所有模块必须调用 common 全局日志工具类；交易、策略敏感数据强制脱敏；日志写入失败静默容错，不阻断主业务

### 5.7 版本迭代与变更标准化
#### 5.7.1 核心规则
- RFC文档版本、代码版本、配置版本、迭代记录**四者对齐**
- 所有新增、修改、重构、废弃操作，必须遵循「RFC更新 → 规范评审 → AI实装 → 版本记录」流程
- 统一全局Changelog格式、迭代说明、版本编号规则
- 废弃功能、过期逻辑统一归档、标记、清理规范

#### 5.7.2 落地约束
- 项目全迭代可追溯、可回滚、可复盘，杜绝隐性变更、无记录修改
- 每次迭代必须更新版本记录，明确变更范围、影响模块

### 5.8 配置与多环境标准化
#### 5.8.1 核心规则
- 所有可变参数、阈值、路径、密钥、策略参数、风控参数**集中配置管理**，禁止硬编码
- 统一区分开发、测试、模拟、实盘多环境，配置完全隔离、互不干扰
- 配置加载优先级、读取规则、更新规则全局统一

#### 5.8.2 落地约束
- AI生成代码禁止出现任何业务参数、密钥、路径硬编码
- 实盘环境与测试环境严格隔离，杜绝参数混用导致交易风险
- 所有配置变更留痕可追溯，适配策略调参、环境切换、复盘迭代

## 6. 全局风险与容错方案
### 6.1 风险矩阵
| 风险类型 | 风险描述 | 概率 | 影响 | 应对方案 | 降级策略 |
|---|---|---|---|---|---|
| 数据风险 | 数据源中断、延迟、格式异常 | 中 | 高 | 多源冗余 + 缓存兜底 | 切换备用源 + 降级数据精度 |
| 策略风险 | 回测参数异常、过拟合实盘失效 | 中 | 高 | 样本外验证 + 参数约束 | 禁止实盘下单 + 日志告警 |
| 交易风险 | 订单失败、市场烈变、滑点扩大 | 低 | 极高 | 风控拦截 + 持仓保护 | 终止下单 + 保资产优先 |
| 报告风险 | 报表生成失败、数据缺失 | 低 | 中 | 历史模板兜底 | 降级输出 + 异常记录 |
| 系统规范风险 | AI生成代码偏离RFC规范 | 低 | 高 | 强制对齐RFC + 人工评审 | 拒绝合并 + 打回重做 |

### 6.2 容错降级原则
- 数据层异常 → 缓存兜底 → 离线备用 → 暂停计算
- 策略层异常 → 参数校验拦截 → 禁止下单 → 告警记录
- 交易层异常 → 风控拦截 → 终止操作 → 持仓保护
- 报告层异常 → 历史模板兜底 → 降级输出 → 异常记录
- AI实装异常 → 强制对齐RFC → 人工评审 → 打回重做

## 9. RFC 规范管理体系
本章节定义 YQClaw Investment 全项目 RFC 治理机制，是所有架构决策、模块设计、接口变更、数据结构变更与 AI 实装任务的统一入口。RFC 采用 **GitHub PR-based RFC + Markdown 文档 + 状态标签** 的轻量治理模式，保证规范可审阅、变更可追溯、实现可验收。

> **说明**：RFC 管理体系作为项目治理的核心规范独立于前文八大标准化体系，八大体系约束工程实现层面，RFC体系约束决策与变更管理层面，两者共同构成 YQClaw 的完整工程规范。

### 10.1 RFC 目的与适用范围
#### 10.1.1 目的
把重要技术决策显式化，避免量化系统在数据口径、接口契约、策略逻辑、交易边界与 AI 生成代码上出现不可追溯的隐性分叉。

#### 10.1.2 适用范围
- 全局架构、模块边界、目录结构、依赖规则与工程规范变更
- 数据模型、数据库 Schema、数据字典、字段口径、数据质量规则变更
- 模块 API、事件协议、错误码、配置项、日志规范与权限模型变更
- 策略、回测、组合、交易、风控、报告等核心业务流程设计
- AI 实装规范、自动化任务边界、代码生成约束与验收标准

#### 10.1.3 不适用范围
- 文档修正、错别字、非语义调整（可直接合并）
- 微小配置变更、紧急补丁（事后补齐 RFC 或变更记录）

### 10.2 RFC 类型定义
| 类型 | 适用场景 | 示例 |
|---|---|---|
| 全局总纲 RFC | 影响全项目架构、规范、治理机制 | RFC-00-001 全局架构总纲 |
| 模块架构 RFC | 定义单一模块职责、接口、数据流与边界 | data 模块数据管道 RFC |
| 接口/契约 RFC | 定义跨模块 API、事件、错误码、权限契约 | portfolio 调仓接口 RFC |
| 数据/Schema RFC | 定义数据表、字段口径、校验与迁移策略 | market_data 表结构 RFC |
| 策略/研究 RFC | 定义投研假设、因子、回测方法、验收指标 | 动量因子研究 RFC |
| 运维/安全 RFC | 定义部署、监控、密钥、权限、备份、回滚 | 日志归档与告警 RFC |
| 废弃/替代 RFC | 废弃旧设计或声明新 RFC 替代关系 | Superseded by RFC-xx-xxx |

### 10.3 生命周期状态机
所有 RFC 必须在元数据中声明状态，状态流转遵循以下状态机：

```
Draft → Review → Accepted → Implemented
   ↓       ↓          ↓
   └──→ Rejected / Withdrawn
                   ↓
              Superseded / Obsolete
```

| 状态 | 含义 | 允许动作 |
|---|---|---|
| Draft | 起草中，方案尚未进入正式评审 | 作者可自由修改，可创建 Draft PR |
| Review | 已进入评论期，等待评审意见 | 评审人评论、作者修订，不得直接进入实装 |
| Accepted | 已被采纳并合并，成为设计依据 | 可创建关联实现任务或代码 PR |
| Implemented | 已完成实装并通过验收 | 记录实现 PR、测试结果、上线/归档状态 |
| Rejected | 评审未通过，不再推进 | 保留拒绝原因，后续新方案另开 RFC |
| Withdrawn | 作者主动撤回 | 保留撤回原因，可重新起草新 RFC |
| Superseded | 已被新 RFC 替代 | 标注"Superseded by RFC-{YY}-{NNN}" |
| Obsolete | 因系统演进自然失效 | 保留归档价值，不再用于新实装 |

### 10.4 编号与文件命名规则
新 RFC 采用按年份与流水号统一编号，文件名格式：

```
RFC-{YY}-{NNN}-{模块名}-{简述}.md
```

示例：

```
RFC-26-001-data-market-data-pipeline.md
RFC-26-014-portfolio-rebalance-engine.md
RFC-26-071-research-argus.md
```

规则：
- `{YY}` 为两位年份（2026年 → 26）
- `{NNN}` 为全项目三位递增流水号，不在模块内重置
- `{模块名}` 使用模块目录名或稳定英文短名（小写）
- `{简述}` 使用小写英文短横线，表达核心主题，避免 new、temp、final 等非语义名称
- 历史 RFC 保留原编号；重命名通过独立迁移 PR 完成

### 10.5 目录结构规则
RFC 文件统一保存在 `docs/rfc/` 下，按模块分目录管理：

```
docs/rfc/

    RFC-00-001-yqclaw-investment-global-architecture.md
    RFC-00-000-rfc-template.md
  01_app/
    RFC-{YY}-{NNN}-app-{简述}.md
  02_common/
  03_data/
  04_knowledge/
  05_portfolio/
  06_strategy/
  07_trading/
  08_research/
    argus/
      RFC-08-001-argus-integration.md
      RFC-2026-071_ARGUS/   # 保留为详细设计参考
  09_reports/
  10_infra/
```

规范目录表达式：`docs/rfc/{模块编号}_{模块名}/`

跨模块 RFC 放入主责模块目录；无单一主责模块时放入 `00_project_overview/`，并在元数据中列明影响模块。

### 10.6 编写与评审流程
RFC 流程采用 **issue → PR** 的轻量闭环：

```
issue → PR(draft) → review comment → merge(accepted) → implemented
```

**流程步骤**：
1. **Issue 提案**：提交背景、问题、影响模块、预期收益与是否需要 RFC
2. **Draft PR**：基于 `RFC-00-000-rfc-template.md` 创建 Markdown RFC，状态为 Draft，补齐元数据、目标、非目标、方案、风险、验收标准
3. **评论期**：进入 Review 后由相关模块 owner、主智能体或指定评审人评论
   - 关键 RFC（数据 Schema、交易、风控、安全、全局架构）：至少 **3 个自然日**
   - 普通 RFC（单模块内部、接口）：建议至少 **1 个自然日**
4. **评审决策**：通过后状态改为 Accepted 并合并；未通过则标记 Rejected 或由作者 Withdrawn
5. **实现关联**：代码实现 PR 必须引用对应 RFC 编号，实现范围不得超出 RFC 的目标与验收标准
6. **验收关闭**：实现完成并通过测试、回测、验收后，将 RFC 状态更新为 Implemented，记录实现 PR、测试结果

### 10.7 审核机制
| RFC 范围 | 必要审核人 | 评论期 |
|---|---|---|
| 文档修正、错别字 | 作者或任一维护者 | 可直接合并 |
| 单模块内部设计 | 模块 owner 或主智能体 | ≥1 个自然日 |
| 跨模块接口、数据流、配置、日志规范 | 相关模块 owner + 主智能体 | ≥1 个自然日 |
| 数据 Schema、策略实盘链路、交易、风控、安全 | 主智能体 + 相关模块 owner | ≥3 个自然日 |
| 全局架构、RFC 流程、模块边界、不可逆迁移 | PascalMao 或授权主智能体 + 关键模块 owner | ≥3 个自然日 |

审核重点：
- 是否解决真实问题，ROI 是否高于治理成本
- 是否保持模块边界清晰，避免循环依赖和跨层硬耦合
- 是否明确数据口径、接口契约、异常降级与验收标准
- 是否给出可执行的实装边界，避免 AI 代码生成自由发挥
- 是否说明风险、回滚、迁移和旧设计兼容策略

### 10.8 变更管理规则
- 已 Accepted 或 Implemented 的 RFC 不做大段覆盖式改写；重大变更必须新开 RFC
- 新 RFC 替代旧 RFC 时，旧 RFC 状态标记为 **Superseded**，显式标注"Superseded by RFC-{YY}-{NNN}"
- 功能自然过期但无直接替代方案时，状态标记为 **Obsolete**，保留归档价值
- RFC、代码、配置、数据库迁移、测试报告必须保持可追溯关联
- 任何接口、Schema、策略逻辑、交易/风控链路变更，都必须**先更新 RFC，再进入实现**

### 10.9 AI 实装关联强制规则
YQClaw Investment 采用 **RFC 先行，AI 实装跟进** 的强约束：

1. 所有代码实装、接口变更、数据库变更、策略逻辑变更、交易/风控链路变更**必须关联 RFC**
2. **无 RFC 不许实装**；紧急修复可先提交最小补丁，但必须在同一工作周期内补齐事后 RFC
3. AI 工具执行任务时，必须在任务说明中引用 RFC 编号、目标章节、允许修改范围、验收标准和禁止事项
4. AI 生成代码不得突破 RFC 约束；发现 RFC 不足时，应先提出 RFC 修订，而不是直接扩大实现范围
5. 实装 PR 必须在描述中写明 `Implements RFC-xxxx` 或 `Refs RFC-xxxx`，并附测试/回测/验证结果
6. 主智能体在合并或交付前必须检查"RFC 目标 — 代码改动 — 验收结果"三者是否一致
