## 1. 执行摘要
本RFC为 YQClaw Investment 项目（AI智能投资交易系统）唯一总架构规范，定义系统定位、整体架构、完整模块划分、依赖关系、数据流转、AI实装规则、开发标准与**全模块八大标准化体系**。项目新增策略研发模块、报告定制模块，统一目录、代码、数据、接口、AI开发、异常容错、版本迭代、环境配置全维度规范。

本文档是所有子模块RFC编写、AI工具自动化开发实装的唯一依据，所有子模块RFC及代码实装必须严格遵循本总纲规范，保障系统全局统一、架构一致、可运维、可复盘、可自动化落地、长期可迭代。

## 2. 背景与动机
### 2.1 业务痛点
当下个人投资存在信息碎片化、投研效率低下、交易决策情绪化、组合风控滞后、策略无法系统化研发回测、缺少标准化监测与定制报告体系，数据不通畅、流程无法闭环、无AI智能辅助决策等核心问题。

同时，传统开发模式无统一工程规范，存在目录杂乱、代码风格不统一、数据口径混乱、接口不标准、AI生成代码差异化严重、无统一容错降级机制、迭代无追溯、配置随意硬编码等问题，导致AI编码工具无法精准落地、模块对接冲突、后期维护迭代成本极高，且量化系统对**数据一致性、流程可追溯、运行可复盘、风险可控**的核心诉求无法满足。

### 2.2 项目定位
打造自主可控，AI Agent全自动驱动的闭环智能投资交易系统，覆盖多品类资产投研、组合管理、策略研发回测、自动化交易、定制化报告、风险控制、投资知识沉淀、数据一体化处理全流程。

完全遵循**RFC先行、规范统一、AI实装跟进**的顶级AI工程化开发模式，配套全维度标准化工程体系，打造个人专属**标准化、自动化、可复盘、高稳健**的智能投资体系。

### 2.3 规范建设价值
通过建立全模块统一标准化体系，彻底解决项目开发杂乱、AI落地偏差、模块耦合、数据不一致、运维无依据、迭代无追溯的核心问题，实现**架构统一、代码统一、数据统一、行为统一、运维统一**，适配多AI工具协同开发，保障系统长期稳定迭代。

## 3. 目标与非目标
### 3.1 核心目标
- [ ] 定义YQClaw Investment系统完整模块化架构，新增策略模块、报告模块，划分清晰模块边界
- [ ] 制定全系统统一RFC编写规范、模块命名、数据、接口、AI实装标准
- [ ] 建立**八大全局标准化体系**，实现全模块工程规范统一
- [ ] 明确模块依赖、数据流转，AI Agent全局调度逻辑
- [ ] 所有子模块RFC必须继承本总纲规范，一一对应、严格对齐
- [ ] 完全适配AI工具全自动实装，零架构偏差、零代码混乱、零规范冲突
- [ ] 实现系统可追溯、可复盘、可容错、可扩展、可运维

### 3.2 非目标
- [ ] 不涉及对外经营性资管、理财服务，仅限个人专业投资使用
- [ ] 不设计冗余架构、不过度技术选型，聚焦投资业务核心，不做无效标准化堆砌
- [ ] 不接入第三方商用运维、日志、监控平台，保持项目轻量化、自主可控

## 4. 全局模块完整架构与职责
### 模块编号&amp;名称&amp;核心职责
00_project_overview：项目全局总架构，统一规范，全局约束、标准化体系定义
01_app：PC/移动端/H5 全端前后端应用与用户交互
02_common：全项目通用金融工具，公共计算、基础能力组件、全局工具封装、统一规范落地载体
03_data：市场数据采集、清洗、Pipeline，标准数据接口、全链路数据标准化落地
04_knowledge：投资理念、投研逻辑、策略知识库、领域知识图谱
05_portfolio：可投池管理、组合构建、持仓分析，调仓、业绩归因
06_strategy：各类量化/主观策略研发，历史回测、参数优化，对接模拟盘与实盘交易
07_trading：交易通道对接、订单管理、实盘/模拟盘执行、交易风控清算
08_research：股票/基金/ETF/Crypto资产研究，多因子评分、机构/卖方/牛散/社区投研、金融舆情
09_reports：资产监测报告、组合风控报告、每日/每周常规监测报表、个性化定制投研报告，自动导出归档
10_infra：系统底座，AI Agent调度、权限，日志、监控、部署运维、底层架构、全局规范落地

### 4.3 全局模块依赖规则
1. 上层模块可调用下层模块能力，下层不可反向调用上层，严格单向依赖
2. 同级模块相互独立，通过标准接口通信，禁止循环依赖、跨模块硬耦合
3. Strategy策略模块依赖数据、知识、组合模块，向下对接Trading交易模块
4. Reports报告模块依赖投研、组合、策略、数据模块，统一生成各类报表
5. 所有模块统一依赖common全局基础能力，统一遵循全局标准化规范

## 5. 全局八大标准化体系（全模块强制遵守）
本章节为**项目最高强制规范**，所有子模块设计、RFC编写、AI代码实装、迭代优化必须100%遵循，禁止私自变更、非标实现。

### 5.1 目录与文件命名标准化
#### 5.1.1 核心规则
- 全局目录、文件夹：纯小写+下划线命名，无驼峰、无中文、无特殊字符、无空格
- 业务代码文件：统一遵循 `{模块名}_{功能名}.py` 语义化命名
- RFC文档文件：统一遵循 `RFC-XX-XXX-模块名-功能简述.md` 标准编号命名
- 配置文件、日志文件、静态资源、脚本文件统一范式，禁止随机命名

#### 5.1.2 落地约束
- AI生成代码、新增目录必须严格对齐全局结构，禁止私自新建层级、乱放文件
- 所有文件命名语义唯一，见名知意，杜绝模糊命名（test、temp、new等非标命名）
- 项目全生命周期目录结构固定，新增功能必须在对应模块内扩展，不破坏全局结构

### 5.2 代码工程结构与风格标准化
#### 5.2.1 核心规则
- 全局严格遵循 Python PEP8 工业级编码规范
- 所有业务模块统一内部结构：`config / core / utils / model / service`，层级统一
- 类、函数职责单一，禁止超大函数、超大类、冗余代码
- 统一注释规范：核心逻辑注释**设计原因+实现逻辑**，而非仅注释代码行为

#### 5.2.2 落地约束
- 所有AI工具生成代码必须统一风格，多AI输出观感、结构、范式完全一致
- 禁止模块自定义编码风格、私自新增非标目录结构
- 核心功能必须补充简洁单元测试，代码可复用、可维护、可迭代

### 5.3 全链路数据标准化（量化核心强制规范）
#### 5.3.1 核心规则
- 金融字段统一：时间、价格、收益率、仓位、市值、波动率、回撤等核心字段**全局唯一命名**
- 数据类型统一：浮点精度、时间格式、空值定义、数值单位全局统一
- 数据流转统一：严格遵循「数据层→知识层→投研层→策略层→交易层→应用层」单向流转
- 数据运维统一：缓存规则、清洗规则、校验规则、降级兜底规则全模块一致

#### 5.3.2 落地约束
- 禁止各模块自定义时间格式、数据精度、统计口径，杜绝数据对账偏差、回测失真
- 所有数据结构新增、修改、废弃必须更新对应RFC，禁止无文档私自改结构
- 量化核心数据强制脱敏、校验，保证数据一致性、准确性、可复盘性

### 5.4 模块接口与调用标准化
#### 5.4.1 核心规则
- 全局统一接口请求、响应、异常返回格式
- 全局统一错误码、状态码、提示文案规范
- 模块间严格分层调用，只允许上层调用下层、同级标准化互通，禁止跨层乱调用、循环依赖

#### 5.4.2 落地约束
- 所有接口必须包含参数校验、异常捕获、日志埋点
- AI生成接口逻辑必须对齐全局契约，禁止自定义返回结构、错误规则
- 所有接口变更必须走RFC迭代，保证全模块对接兼容

### 5.5 多平台AI实装开发标准化
#### 5.5.1 核心规则
适配 OpenClaw、Claude Code、Gemini 全AI平台，统一开发规则：
- AI权限规则统一：明确可直接执行、需先询问、绝对禁止的操作
- AI代码生成范式统一：日志埋点、异常捕获、注释、单元测试、代码简洁度统一
- AI任务拆分统一：单指令单模块、单功能，禁止跨模块大范围修改
- AI迭代逻辑统一：**先更新RFC规范，再落地代码实现**

#### 5.5.2 落地约束
- AI禁止自由发挥、过度设计、冗余编码、偏离RFC设计
- AI禁止硬编码参数、密钥、路径等可变配置
- 所有AI开发成果必须与RFC文档1:1映射，无偏差落地

### 5.6 异常、容错与降级体系标准化
#### 5.6.1 核心规则
- 异常分类标准化：统一分为数据异常、策略异常、交易异常、业务异常、系统异常、AI执行异常
- 异常处理标准化：统一捕获范式、报错输出、堆栈打印、日志记录规则
- 降级策略标准化：全局统一兜底逻辑，数据兜底、策略暂停、交易阻断、报告降级、服务容错

#### 5.6.2 落地约束
- 所有核心业务必须具备降级方案，异常不阻断核心流程或可控阻断
- 交易、策略等高风险场景，异常优先风控、优先保资产安全
- 所有异常必须可追溯、可复盘、可定位根因

### 5.7 版本迭代与变更标准化
#### 5.7.1 核心规则
- RFC文档版本、代码版本、配置版本、迭代记录**四者对齐**
- 所有新增、修改、重构、废弃操作，必须遵循「RFC更新 → 规范评审 → AI实装 → 版本记录」流程
- 统一全局Changelog格式、迭代说明、版本编号规则
- 废弃功能、过期逻辑统一归档、标记、清理规范

#### 5.7.2 落地约束
- 项目全迭代可追溯、可回滚、可复盘，杜绝隐性变更、无记录修改
- 每次迭代必须更新版本记录，明确变更范围、影响模块

### 5.8 配置与多环境标准化
#### 5.8.1 核心规则
- 所有可变参数、阈值、路径、密钥、策略参数、风控参数**集中配置管理**，禁止硬编码
- 统一区分开发、测试、模拟、实盘多环境，配置完全隔离、互不干扰
- 配置加载优先级、读取规则、更新规则全局统一

#### 5.8.2 落地约束
- AI生成代码禁止出现任何业务参数、密钥、路径硬编码
- 实盘环境与测试环境严格隔离，杜绝参数混用导致交易风险
- 所有配置变更留痕可追溯，适配策略调参、环境切换、复盘迭代

### 5.9 全局日志统一管理规范（原有规范整合强化）
#### 5.9.1 日志存储目录规范
项目采用**根目录集中式日志管理**，所有模块日志统一存放于 workspace-yquant/logs/ 目录，按模块分文件夹隔离，禁止自定义日志路径、禁止代码内直接打印输出：
- logs/system/ 系统全局日志（启动、报错、服务监控、AI调度记录）
- logs/app/ 应用端交互、操作、页面访问日志
- logs/common/ 公共工具、通用函数执行日志
- logs/data/ 数据采集、清洗、入库、Pipeline运行日志
- logs/knowledge/ 知识库加载、规则匹配、知识调用日志
- logs/portfolio/ 组合构建、调仓、持仓分析、业绩归因日志
- logs/strategy/ 策略初始化、参数优化、回测运行、策略信号日志
- logs/trading/ 订单创建、撤单、成交、交易风控、清算日志
- logs/research/ 投研分析、因子计算、舆情解析、资产评分日志
- logs/reports/ 报告生成、数据渲染、导出、归档异常日志

#### 5.9.2 日志命名、格式、级别规范
- 命名格式：`{模块名}_{YYYYMMDD}.log`，按天自动切割
- 统一输出格式：`[年-月-日 时:分:秒] [模块名] [日志级别] [日志内容]`
- 全局统一级别：DEBUG、INFO、WARN、ERROR、ORDER（交易专属）

#### 5.9.3 AI实装强制约束
1. 全项目禁止使用print输出，所有运行记录、异常、业务行为必须通过统一日志工具输出
2. 所有模块必须调用 common 全局日志工具类，禁止自定义日志逻辑
3. 关键业务节点必须留存INFO级别日志，报错必须留存ERROR级别日志
4. 交易、策略敏感数据强制脱敏，日志写入失败静默容错，不阻断主业务

## 6. 全局风险与容错方案
- 数据风险：数据异常→缓存兜底→离线备用→暂停数据计算
- 策略风险：回测参数异常→参数校验拦截→禁止实盘下单→日志记录告警
- 交易风险：交易异常→风控拦截→终止下单→保护持仓→留存交易日志
- 报告风险：报表生成失败→保留历史模板→降级输出基础报告→记录异常
- 系统规范风险：AI生成代码偏离规范→强制对齐RFC→人工评审修正

## 7. 验收标准（新增标准化验收维度）
### 7.1 功能验收

- 全模块业务流程闭环、数据通畅、运行稳定、风险可控

### 7.2 标准化验收（新增强制项）
- 全模块严格遵循八大全局标准化体系，无任何非标实现
- 目录、文件、代码、数据、接口、日志、配置全部统一规范
- AI工具可完全依据本RFC+子模块RFC**零偏差全自动开发**
- 项目全流程可追溯、可复盘、可运维、可迭代

## 8. 落地实施计划

3. 存量迭代、新增功能、模块重构优先适配标准化规范，逐步统一全项目体系
4. 所有规范变更必须通过总纲RFC迭代更新，全局同步生效
