### 5.2bis 持久化策略（Persistence Strategy）
如本 RFC 涉及任何需要跨会话、跨进程、审计或回放的数据，必须在此说明持久化边界；如不涉及，写“无持久化需求”并说明原因。

| 数据类别 | 存储介质 | 命名规则（库/集合/表/路径） | 生命周期/保留期 | 隐私级别与跨边界规则 |
|---|---|---|---|---|
| 示例：任务元数据 | MongoDB/PostgreSQL/文件/无 | `{module}_{purpose}` 或项目规范命名 | 创建、更新、归档、删除策略 | L0/L1/L2/L3；是否允许跨库/跨项目引用 |

必须覆盖：
- 存储选型：为什么需要持久化，为什么选择该介质，哪些数据明确不落盘。
- 命名与边界：库、集合/表、文件路径或对象 key 的命名规则，以及跨库、跨项目、跨家庭成员边界。
- 生命周期：创建、更新、归档、TTL、删除、备份与恢复要求。
- 隐私与最小化：字段级隐私分级，L2/L3 数据的脱敏、匿名化或禁止入库策略。
- 审计与可追溯：需要记录哪些操作、来源、版本、任务 ID 或操作者信息。
- 回滚与迁移：schema 变更、数据迁移、失败回滚和兼容旧数据的原则。

## 6. AI实装规范（AI Implementation Rules）
### 6.1 必须执行
- 单指令只做一件事，使用相对路径
- 代码简洁可控，命名语义化
- 核心逻辑补充单元测试
- 所有变更保留可追溯记录

### 6.2 先询问再执行
- 修改数据结构，新增第三方依赖
- 变更对外接口、影响现有业务逻辑
- 涉及密钥、权限、配置变更

### 6.3 绝对禁止
- 硬编码敏感密钥与凭证
- 随意删除历史文件与目录
- 无方案大范围重构代码
